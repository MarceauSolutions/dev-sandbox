# Workflow: Building LinkedIn/GitHub Presence

## Overview
Strategy for documenting and showcasing your dev-sandbox projects to attract consulting clients and job opportunities.

## GitHub Strategy

### Repository Structure
Your projects should be public on GitHub with clear READMEs:

| Project | Repo Name | Target Audience |
|---------|-----------|-----------------|
| Fitness Influencer | `fitness-influencer-ai` | Fitness creators, potential users |
| MCP Aggregator | `mcp-aggregator` | AI developers, potential partners |
| Lead Scraper | `local-business-leads` | Other consultants (showcase capability) |
| Time Blocks | `time-blocks-calendar` | Productivity enthusiasts |

### README Template for Each Project
```markdown
# Project Name

One-line description of what it does.

## Demo
[Screenshot or GIF of the tool in action]

## Features
- Feature 1
- Feature 2
- Feature 3

## Tech Stack
- Python / FastAPI / PostgreSQL
- APIs: Google Calendar, xAI, etc.
- Deployed on: Railway / Local

## Quick Start
[Instructions to run locally]

## Results
- "Reduced client admin time by 80%"
- "Processing 100+ leads/week"
- Concrete metrics

## Author
William Marceau Jr. | [LinkedIn](your-link) | Automation Consultant
```

### What NOT to Make Public
- API keys (use .env.example instead)
- Client-specific data
- Proprietary algorithms (MCP scoring, etc.)
- Personal assistant configurations
