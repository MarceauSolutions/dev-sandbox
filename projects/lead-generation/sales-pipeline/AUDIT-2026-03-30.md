### 2. Multiple Conflicting Data Sources
- `phone_blitz_2026-03-24.json` — Raw phone leads
- `inperson_route_2026-03-25_prepared.json` — In-person route prep
- `naples_phone_leads_2026-03-23.json` — Initial lead list
- `session-history.md` — **Manually written, claimed actions that didn't happen**
- `outreach_log` table — **Orphan records disconnected from deals**
- `deals` table — **THE ONLY SOURCE OF TRUTH** (but not being used consistently)

### 3. Call List Generated from Mixed Sources
Today's `call_list_2026-03-30.csv` pulled Von & Co from legacy JSON files, not the verified database. The notes "Rachel interested, email sent" came from session-history.md, not verified actions.

### 4. Timing Errors in Outreach Logging
Von & Co shows:
- 15:37:00 — "Follow-up email" logged **BEFORE** the visit
- 18:38:31 — "In-person visit" logged **AFTER**

This means the system logged a "follow-up to a visit" before the visit actually happened.

## Corrective Actions Taken

### Database Cleanup
1. Delete all orphan outreach records (no deal_id)
2. Properly add or remove leads from deals table based on verified outcomes
3. Mark leads as "Closed Lost" where we have no legitimate relationship

### Data Source Consolidation
1. Archive legacy JSON files to `archive/legacy-2026-03-25/`
2. ALL future call lists MUST come from `deals` table only
3. Remove or disable any scripts that pull from JSON files

### Call List Generation Fix
1. Delete contaminated `call_list_2026-03-30.csv`
2. Ensure `morning_call_sheet.py` only pulls from verified deals with deal_ids
3. Add validation check: "Has this lead been ACTUALLY contacted?" before including

### Process Changes
1. NEVER log outreach before it happens
2. Always verify database state before generating scripts
3. If session-history.md conflicts with database — database wins
